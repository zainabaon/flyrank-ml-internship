01 · Introduction

The decision this work supports

Content teams cannot review every page on a site for refresh. Given limited reviewer capacity, the real question is not "is this page declining?" in isolation — it's which pages should be reviewed first, given that only a handful can be checked in any given cycle.

This project ranks pages by refresh priority so a reviewer with limited time works from the top of a queue down, rather than guessing. The cost of a wrong call is asymmetric: flagging a healthy page wastes a reviewer's time, while missing a genuinely declining, high-traffic page means it keeps losing visibility unreviewed.

This is a decision-support tool, not an automated fix. No claim is made that refreshing a flagged page causes it to recover — that would require a causal experiment this data cannot provide.

02 · Data

What this is built on

This project uses the FlyRank internship warehouse release (Hugging Face, gated, build flyrank_pseudonymized_warehouse_release_v20260703), specifically:

  • fact_content_daily_performance — daily × client × content grain, month-partitioned (developed against month=2026-03, a mid-panel month; the final month is treated as a sealed test window and was not used for label logic)
  • dim_content and dim_clients — join keys and client-level context only, never used as model features
  • The small anonymized starter dataset (content_refresh_anonymized.csv, 30,000 rows) — used for iteration, baseline development, and the modeling comparisons shown in the Results section below

Excluded, deliberately: FlyRank's own product decision fields (health_score, priority_score, action_type, refresh flags) are not shipped in this release and were never used as features or labels — using them would only teach a model to copy an existing rule rather than discover real signal. No client names, domains, raw URLs, or raw queries appear anywhere in this data or in this paper; all identifiers are pseudonymized hash IDs used for grouping only.

03 · Methodology

Task, label, features, and validation

Task type

This is a scoring / ranking problem built on top of a binary classifier. The deliverable is not a single yes/no decision on one page — it's an ordered queue of all pages, ranked by predicted refresh priority, so a reviewer works top-down within their capacity.

Label / proxy

The working label is is_declining_label = (trend_direction == "down") — a same-window proxy, not a true future outcome. This is a known and disclosed limitation (see Limitations below); a stronger future-window label (prior-window features predicting decline over a following window) is the natural next iteration.

Baseline (Week 4)

A transparent hand-written rule: flag a page when it is both stale (180+ days since last update) and visible (500+ impressions in the trailing 90 days). The score is the page's raw impressions, so among qualifying pages, highest-traffic pages rank first.

Before trusting this rule, two of its underlying signals were checked directly against the data:

  • Staleness vs. decline rate — verdict: MIXED. Decline rate rose from the <30-day bucket through the 90–180-day bucket, but did not continue rising monotonically past 180 days, and the oldest bucket had too few rows (n=5) to trust. Staleness alone is a weak signal past 180 days; it is only useful combined with visibility, which the rule already does.
  • CTR vs. position tier — verdict: CONFIRMED. Mean CTR fell steadily from top-ranked to lowest-ranked tiers, confirming CTR should only be compared within the same position tier, not across the dataset as a whole.

Model (Week 5)

Three classifiers were trained and compared: Logistic Regression, a depth-limited Decision Tree, and a Random Forest, using six observed, pre-decision features: content_age_days, days_since_last_update, impressions_90d, avg_position, ctr, word_count.

Validation design

Client-grouped holdout split (75/25) — pages from the same client are never split across train and test, verified directly (zero client overlap). This matters because a random row split would let a model partially memorize a client's own baseline traffic patterns rather than learning generalizable signal, inflating apparent performance.

Leakage audit

  • All six features are observed at the same snapshot as the label — not future-derived.
  • trend_pct (which directly determines the label) is deliberately excluded from every feature set.
  • No FlyRank product decision fields were available or used.
  • Client overlap between train and test is checked and confirmed at zero under the grouped split.
04 · Results

Model vs. baseline, same split, same metric

All methods were evaluated on the identical client-holdout test set (n=7,115 rows, 8 held-out clients) using Precision@50 — of the top 50 ranked pages, how many were genuinely in the "declining" bucket. This metric was chosen over accuracy because the real use case is a fixed-size review queue, not a judgment across the entire dataset.

Logistic Regression was the only method to beat the baseline on Precision@50 (0.660 vs. 0.560). Random Forest had the highest AUC (0.597) but did not exceed the baseline on the top-K ranking metric that actually reflects the review-queue use case — a reminder that added model complexity does not automatically translate into a better ranked output.

What the errors look like

Feature importance from the Random Forest model shows impressions_90d (0.30), avg_position (0.23), and content_age_days (0.20) as the strongest signals — while days_since_last_update, the core signal the hand-written baseline relies on most, ranked lowest (0.05). This independently confirms the MIXED verdict found in the Week-4 signal audit: staleness alone is a weak predictor once other signals are available.

All models scored only modestly above chance (AUC 0.54–0.60) on the held-out clients — noticeably lower than same-window, non-holdout performance seen earlier in the internship pipeline (~0.75). This gap suggests the relationship between these features and same-window decline is noisier across different clients than within any single client's own history, and reinforces that a genuine future-window label (rather than a same-window proxy) is likely necessary for a more decision-useful, less noisy system.

05 · Limitations & honest framing

What this can and cannot claim

What is supported: on this observed sample and this validation split, Logistic Regression showed a measurably higher Precision@50 than a transparent hand-written rule. This is decision-support evidence for prioritizing a limited review queue — nothing more.

What is not supported:

  • No claim that refreshing a flagged page causes it to recover — that requires a causal experiment, not observational ranking.
  • No claim of having identified a Google ranking factor, or of predicting how any search algorithm works.
  • The working label is a same-window proxy (trend_direction == "down"), not a true future outcome — results describe agreement with a current-state bucket, not a validated forecast.
  • Results are drawn from a single client-holdout split on a bounded sample; they are directional, not a guarantee that the same gap holds on new clients, new time windows, or the full warehouse.
